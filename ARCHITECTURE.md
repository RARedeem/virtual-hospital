# 私人家族虚拟医院 — 系统架构文档

> 版本：v2.0（双盲交叉验证架构）
> 前序：v1.0 为单流程翻译三明治管道，已验证跑通，作为流程 B 纳入本架构。
> 日期：2026-06-14
> 状态：架构设计完成，待实现

---

## 一、系统定位

私人家族健康档案 + AI 循证健康评估系统。

- 本地部署，数据不出家门
- 对标三甲全科的循证参考能力
- 无医生角色、无诊疗行为、无商业用途
- 仅供成员A本人及家属使用
- **系统输出不构成医学诊断，不替代专业医疗**

---

## 二、硬约束（不可违反）

### 约束 A：模型来源
全链路模型只能使用非中国大陆机构开发的模型。

| 用途 | 模型 | 开发机构 | 合规 |
|------|------|----------|------|
| 流程 A 问诊+推理 | llama3.3:70b 或 llama4:16x17b | Meta (美国) | ✓ |
| 流程 B 循证推理 | meditron:70b | EPFL (瑞士) | ✓ |
| 翻译（中↔英） | gemma4:31b（自定义 translator） | Google (美国) | ✓ |
| 向量检索 | nomic-embed-text:v1.5 | Nomic AI (美国) | ✓ |

禁止：Qwen / DeepSeek / GLM / BGE 及其他中国大陆机构模型。

### 约束 B：指南来源（分流程适用）
- **流程 B**：锁死在国际权威指南（WHO / ADA / AHA / NICE / EAU / MDS）。
  不可使用中国大陆机构（卫健委/中华医学会/药监局等）发布的指南。
  此约束写入数据库 CHECK 约束和 ingest.py 黑名单，双重防线。
- **流程 A**：可引入国内指南用于 A2 推理阶段。
  这是架构设计决策，目的是制造证据层异质性（详见第五节）。
- **会员原始数据**：会员上传的实体医院检验报告（含大陆医院出具的）
  属个人医疗事实，不受约束 B 限制，可正常入库。

## 三、架构总览

```
                    ┌──────────────────────────────────┐
                    │           患者（用户）            │
                    └──────────┬───────────────────────┘
                               │
                    ┌──────────▼───────────────────────┐
                    │    流程 A1：问诊接诊（llama）     │
                    │    无指南介入，纯问诊逻辑         │
                    │    交互式收集症状/病情信息         │
                    └──────────┬───────────────────────┘
                               │
                       症状数据打包封装
                               │
                    ┌──────────┴───────────┐
                    ▼                      ▼
         ┌─────────────────┐    ┌─────────────────────┐
         │  流程 A2：推理   │    │  流程 B：推理        │
         │  llama           │    │  meditron            │
         │  国内指南 + RAG  │    │  国际指南 + RAG      │
         │  + 患者历史数据  │    │  + 患者历史数据      │
         │  + 规则引擎      │    │  + 规则引擎          │
         └────────┬────────┘    └──────────┬──────────┘
                  │                         │
                  │    结论 A（独立）         │  结论 B（独立）
                  │                         │
                  └──────────┬──────────────┘
                             ▼
                  ┌──────────────────────┐
                  │  呈现层（代码逻辑）   │
                  │  非 LLM，纯模板格式化 │
                  │  并排展示 A、B 结论   │
                  │  标明各自证据来源     │
                  │  + 醒目声明条款       │
                  └──────────┬───────────┘
                             ▼
                  ┌──────────────────────┐
                  │   最终报告 → 用户     │
                  └──────────────────────┘
```

**核心设计原则：背靠背双盲。**
流程 A 和流程 B 各自独立推理，互不知道对方的结论。
最终报告并排呈现两份结论，判断权归人，系统不仲裁。

## 四、流程详述

### 4.1 流程 A1：问诊接诊

| 项目 | 说明 |
|------|------|
| 执行模型 | llama3.3:70b 或 llama4:16x17b（待定） |
| 指南介入 | **无**。问诊阶段不引入任何指南 |
| 职责 | 与患者交互式问诊，收集症状、病史、用药、家族史等 |
| 产出 | 结构化症状数据包（供 A2 和 B 共同消费） |
| 设计依据 | 现实中医生接诊时不翻指南，先把信息问全，指南在推理时才介入 |

**A1 的核心约束**：问诊收集的信息必须足够支撑后续两个流程的循证推理。
如果信息不足，A1 应向患者追问，而非让 A2/B 在信息缺失下硬推。

### 4.2 流程 A2：国内指南循证推理

| 项目 | 说明 |
|------|------|
| 执行模型 | llama（同 A1，复用会话上下文） |
| 指南来源 | 国内权威指南（卫健委/中华医学会等） |
| 输入 | A1 打包的症状数据 + RAG 检索的国内指南 + 患者历史数据 |
| 产出 | 结论 A（含引用的国内指南来源） |
| 显存管理 | 推理完成后释放显存（keep_alive=0），为流程 B 腾空间 |

### 4.3 流程 B：国际指南循证推理

| 项目 | 说明 |
|------|------|
| 执行模型 | meditron:70b |
| 指南来源 | 国际权威指南（WHO/ADA/AHA/NICE/EAU/MDS），约束 B 锁死 |
| 输入 | A1 打包的症状数据（**不含 A2 的结论**）+ RAG 检索的国际指南 + 患者历史数据 |
| 产出 | 结论 B（含引用的国际指南来源） |
| 显存管理 | 等 A2 释放显存后再加载（串行调度） |

**双盲关键**：B 接收的是 A1 的原始症状数据，不是 A2 的推理结论。
A2 的结论对 B 完全不可见，确保两条推理路径独立。

### 4.4 翻译层

| 方向 | 模型 | 位置 |
|------|------|------|
| 中文 → 英文 | translator-zh-en（基于 gemma4:31b） | A1 产出 → 英文，供 A2 和 B 的 RAG 检索 |
| 英文 → 中文 | translator-en-zh（基于 gemma4:31b） | A2/B 英文结论 → 中文，供最终报告 |

### 4.5 呈现层（原流程 C，已简化为代码逻辑）

**设计决策**：取消独立的第三模型仲裁节点。

原设计中流程 C 是一个 LLM，负责对比 A、B 结论并设阈值仲裁。
经架构评审（证伪推演），发现三个不可解问题：
1. 两段自然语言结论无法稳定量化"相似度"
2. 第三模型不比 A、B 更权威，属"外行仲裁内行"
3. 阈值设高永远回流、设低垃圾放行

**替代方案**：呈现层由纯代码实现（模板/格式化），不消耗显存，确定性输出。

职责：
- 将结论 A 和结论 B **并排展示**，各自标明证据来源
- 一致的地方标注"双模型共识"
- 不一致的地方标注"存在分歧，建议就医确认"
- **不判断谁对谁错，不做医学仲裁，判断权归人**
- 附加醒目声明条款（见第七节）

## 五、异质性设计（双盲的灵魂）

双盲之所以有效，前提是两个推理者的错误不相关。
本架构在两个维度制造异质性：

### 5.1 模型层异质性
| 流程 | 模型 | 训练取向 |
|------|------|----------|
| A | llama | 通用强推理，擅长理解和遵循复杂指令 |
| B | meditron | 医学文献微调，擅长消化医学证据 |

两个模型的训练语料、微调方向不同，认知盲区不同，
对同一份症状数据可能产生不同的推理路径和结论——
这是双盲要捕捉的信号。

### 5.2 证据层异质性
| 流程 | 指南来源 | 特征 |
|------|----------|------|
| A | 国内指南 | 可能更贴合中国人群流行病学特征 |
| B | 国际指南 | 循证等级高、多中心 RCT 支撑 |

**设计决策记录**：
- 证据层异质性经过架构评审中的三轮质疑：
  1. "国内指南会污染流程 B" → 已用双盲数据流（B 只拿 A1 症状数据）解决
  2. "输出对用户不诚实" → 已用路一（全透明并列呈现、标明来源）解决
  3. "买到的是噪声不是信号" → 部分成立，但全透明呈现下用户可自行判断
- 最终决策：引入国内指南到流程 A，但输出端**必须**标明每条结论的指南来源

### 5.3 异质性不正确的放法（设计禁区）
- 不可让 A、B 看到对方的结论（破坏双盲）
- 不可在 B 的输入里混入国内指南的内容（破坏约束 B）
- 不可在输出里隐瞒某条结论的来源（破坏对用户的诚实）

## 六、显存调度

本系统在消费级双卡硬件上运行，显存是硬约束。

### 6.1 核心规则
```
OLLAMA_KEEP_ALIVE=0  （多模型串行管道必须，用完立刻释放）
```

### 6.2 串行调度顺序
```
A1 问诊:  llama 加载 → 问诊完成 → 释放
翻译:     gemma4 加载 → 中译英 → 释放
A2 推理:  llama 加载 → 推理+RAG → 产出结论 A → 释放
B  推理:  meditron 加载 → 推理+RAG → 产出结论 B → 释放
翻译:     gemma4 加载 → 英译中 → 释放
呈现:     纯代码，不消耗显存
```

### 6.3 教训记录（2026-06-14 事故）
OLLAMA_KEEP_ALIVE 曾被设为 24h（骨架默认值），导致模型累积占满显存，
连续触发 /assess 500 错误。用户早已交代 keep_alive=0 但未落实到配置。
**机制：关键运行参数交代后，必须当场核实落地到具体文件第几行。**

### 6.4 显存红线
llama4:16x17b（67GB）在未论证其与评估管道的显存关系前，
不可同时加载。避免挤爆评估管道的显存空间。

## 七、声明条款（报告必须包含）

每份评估报告必须醒目展示以下两条声明：

> **声明一**：本系统基于人工智能模型对健康数据进行循证参考分析，其输出内容不构成医学诊断、医疗建议或处方意见。任何健康相关决策（包括但不限于用药、手术、延迟就医）均应以具有执业资格的医疗专业人员的当面判断为准。使用本系统不构成医患关系，亦不可替代、推迟或取消任何必要的医疗就诊。

> **声明二**：本系统运行于本地消费级硬件环境，受算力与显存资源限制，单次评估的推理过程可能持续数分钟至三十分钟不等。推理期间请勿关闭页面或中断服务。系统响应时间不代表评估质量，较长的处理时间属正常运行状态。

此外，报告正文中每条结论必须标注其证据来源：
- 来自国际指南的标注为 `[国际指南: ADA 2026]` 等
- 来自国内指南的标注为 `[国内指南: 卫健委 xxx]` 等
- 双模型一致的标注为 `[双模型共识]`
- 存在分歧的标注为 `[存在分歧，建议就医确认]`

## 八、双轨评估（每个流程内部）

每个流程（A2 和 B）内部均采用双轨评估架构：

### 8.1 确定性规则引擎（第一轨）
基于硬编码阈值的确定性判断（rules_engine.py + extractor.py）。
规则命中作为**既定事实**注入推理上下文，LLM 不得推翻。

示例：
- HbA1c ≥ 6.5% → 达糖尿病诊断标准 [ADA 2026]
- 收缩压 ≥ 140 mmHg → 1 期高血压 [AHA/ACC 2017 HTN]

### 8.2 LLM 循证推理（第二轨）
基于 RAG 检索指南 + 患者数据的 LLM 自由推理。
推理结果与规则引擎命中合并后形成最终结论。

## 九、知识库架构

### 9.1 摄取管道
```
指南 PDF → Docling（结构化解析，保留章节/表格）
         → 按章节切片（parser.py，SectionHeaderItem 识别）
         → URL-section 过滤（排除伪标题）
         → nomic-embed 向量化 → pgvector 存储
```

### 9.2 检索过滤
retrieve_guidelines 的 WHERE 子句排除非正文章节：
```sql
AND c.section !~* '(reference|bibliography|acknowledg|abbreviation|
    conflict of interest|appendix|disclaimer|publication info)'
```

### 9.3 当前知识库状态（国际指南，流程 B 用）
| citation_id | org | chunks | 状态 |
|-------------|-----|--------|------|
| ADA 2026 | ADA | 2663 | ✓ Docling 摄取 |
| AHA/ACC 2017 HTN | AHA/ACC | 341 | ✓ |
| EAU 2026 LUTS | EAU | 474 | ✓ Docling 重摄取，0 Preamble |
| MDS | MDS | 60 | ✓ |

### 9.4 待建知识库（国内指南，流程 A 用）
需摄取国内权威指南，存入独立的知识库分区或用 source 标记区分。
指南选取标准：国家级权威机构发布，循证等级高。

## 十一、架构评审记录（证伪日志）

本架构经过以下质疑和修正，记录在案供日后回溯：

### 质疑 1：伪独立的交叉验证比没有更危险
- 提出者：Claude（架构评审）
- 内容：如果 A、B 训练语料相近、又用同一份指南，犯同样的错却盖上
  "双重验证"的章，制造虚假可信度
- 结论：**接受**。异质性必须在模型层和证据层同时保证（见第五节）

### 质疑 2：流程 C 的仲裁资格
- 提出者：Claude
- 内容：第三模型凭什么比 A、B 更权威？外行仲裁内行
- 结论：**接受**。取消流程 C 作为独立模型，改为纯代码呈现层。
  只呈现不评价，判断权归人

### 质疑 3：一致 ≠ 正确
- 提出者：Claude
- 内容：A、B 100% 一致不代表医学正确，必须在输出上诚实
- 结论：**接受**。报告必须写明"两个AI模型的共识参考，不是医学诊断"

### 质疑 4：国内指南污染流程 B
- 提出者：Claude
- 内容：国内指南从流程 A 流入，可能污染 B 的独立性
- 回应：用户设计了双盲数据流——B 只接收 A1 的原始症状数据，不接收 A2 的结论
- 结论：**撤回质疑**。数据流设计已隔离，污染不成立

### 质疑 5：输出端对用户不诚实
- 提出者：Claude
- 内容：系统宣称"基于国际指南"，但报告混入国内指南结论
- 回应：用户选择路一——全透明并列呈现，标明每条结论的来源
- 结论：**解决**。不隐瞒，全摊开，承诺与输出一致

### 质疑 6：相似度阈值无法稳定定义
- 提出者：Claude
- 内容：两段自然语言结论怎么量化相似度、阈值怎么定
- 回应：取消流程 C 的量化比对，改为并排呈现
- 结论：**问题消除**。不量化、不设阈值、不回流，直接呈现

## 十二、设计原则（贯穿全项目）

1. **数据不出家门**：全部本地部署，无任何外部 API 调用
2. **不替代医生**：输出是参考，不是诊断；系统越可信越要醒目声明
3. **对用户诚实**：来源全标、分歧全露、不藏不掖
4. **约束焊死**：约束 A/B 不可被任何"聪明的理由"绕过（证据层异质性除外，已评审通过）
5. **文件化**：关键决策、参数、教训写进文档，不靠记忆
6. **证伪优先**：架构的价值在于禁得住推敲，冷水比表扬有用
7. **真火箭检验**：设计层证伪用逻辑，实现层证伪用真数据跑真失败
