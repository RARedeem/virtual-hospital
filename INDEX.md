# 文档溯源子系统 — 交付索引（INDEX）

> 版本：v1.0 ｜ 日期：2026-06-16 ｜ 状态：代码 + 文档完成，待 Claude Code 落地
> 定位：本文件是**文档溯源子系统**整套交付物的入口。
>       它是虚拟医院项目的一个支撑子系统，非主线。主线入口仍是项目根的 README.md / HANDOFF.md。

文档溯源子系统解决一件事：对设计文档（ARCHITECTURE.md、声明条款等）做**段落级变更溯源**——
每个文本单元可追溯修改史、修编人、版本号，并支持语义检索。
与医疗知识库物理隔离，与 git 互补（git 管文件快照，本系统管段落级血缘 + 语义检索）。

## 一、整套交付物（10 个文件，四层 + 代码 + 拓扑）

```
provenance-system/
├── INDEX.md                      ← 你在这里（入口）
│
├── 【架构层】 ARCHITECTURE-PROVENANCE.md   （在项目根，非本目录）
│       为什么做、约束是什么、设计决策记录
│
├── 【模块设计层】 MODULE-DESIGN.md
│       每个模块为什么这么设计、内部怎么运转、关键决策
│
├── 【契约层】 DATA-INTERFACE-CONTRACT.md
│       数据结构（两表字段/约束/不变量）+ 函数签名 + 错误行为
│
├── 【落地层】 README.md
│       怎么跑、落地对齐点（Claude Code 须核对）
│
├── 【总控】 SYSTEM-TOPOLOGY.html
│       全局系统拓扑（上帝视角，交互式，浏览器直接打开）
│
├── db/
│   └── 03_provenance_schema.sql  ← schema + 两表 + 四索引
│
└── provenance/
    ├── splitter.py               ← md → 段落级单元（纯函数）
    ├── embedder.py               ← nomic 768 维向量化封装
    ├── lineage.py                ← 血缘判定 0.85 阈值（纯函数）
    ├── provenance_ingest.py      ← 写入主流程（方案B手动触发）
    └── query.py                  ← 血缘历史 + 语义检索
```

注：架构层文档 `ARCHITECTURE-PROVENANCE.md` 是本子系统的设计源头，位于项目根目录
（与 ARCHITECTURE.md 并列），是本交付的前序，不在本目录内但属于同一文档体系。

## 二、按角色推荐的阅读顺序

不同来人，进入路径不同。各取所需，不必全读。

### A. 第一次接触本子系统 → 先看全局，再钻细节
1. **SYSTEM-TOPOLOGY.html**（开浏览器，点节点）——10 分钟建立全局心智图，
   看清文档溯源在整个虚拟医院里的位置、与谁共享基础设施、被哪条隔离边界框住。
2. **本 INDEX**（你正在读）——知道有哪些文件、怎么串。
3. **ARCHITECTURE-PROVENANCE.md**（项目根）——理解"为什么要做、向量化的真正落点"。

### B. 要动手落地（Claude Code）→ 直奔可执行路径
1. **README.md** 第"落地步骤"+"落地对齐点"——5 个对齐点必须先核对
   （尤其 psycopg 版本、DATABASE_URL、OLLAMA_HOST 容器内网地址）。
2. **db/03_provenance_schema.sql**——先建 schema。
3. 跑 `provenance_ingest.py` 首次摄取 → `query.py` 验证。
4. 回测（README 末"验证状态"未完成项）：用 ARCHITECTURE.md 的 git 历史
   还原"声明条款进化史"，校准 0.85 阈值。

### C. 要改代码、怕改坏血缘语义 → 先读契约
1. **DATA-INTERFACE-CONTRACT.md**——改任何函数前先读，尤其"关键不变量"
   四条（一条血缘链至多一个 current、version 连续递增、parent 指向前版、旧版永不物理删）。
2. **MODULE-DESIGN.md** 对应模块节——理解该模块"为什么这么设计"，
   避免改掉一个看似多余、实则有意的决策（如 lineage 的一对一贪心匹配）。
3. 改完同步更新契约文档签名 + 拓扑图数据（见第四节）。

### D. 只想理解某一个模块 → MODULE-DESIGN.md 第二节对应小节
五个模块各有独立小节，讲职责 + 内部运转 + 关键决策点，互不依赖可单看。

## 三、四层文档的分工（为什么是四层而非一份）

| 层 | 文件 | 回答的问题 | 读者 |
|----|------|-----------|------|
| 架构层 | ARCHITECTURE-PROVENANCE.md | 为什么做、约束、设计决策 | 决策者、评审 |
| 模块设计层 | MODULE-DESIGN.md | 每个模块怎么运转、为何这么设计 | 维护者、改代码者 |
| 契约层 | DATA-INTERFACE-CONTRACT.md | 数据/接口长什么样、错怎么报 | 落地者、调用方 |
| 落地层 | README.md | 怎么跑、对齐哪些点 | Claude Code |
| 总控 | SYSTEM-TOPOLOGY.html | 全局怎么咬合（上帝视角） | 所有人，第一站 |

分层的意义：各层变更频率不同。架构层最稳定（设计定稿少动），落地层最易变
（环境一变就改）。分开后改落地细节不污染架构论证，符合项目"文件化、分层钉死"的纪律。

## 四、易碎一致性提醒（改动时必读）

这套交付里有三处"手工对齐、易漂移"的一致性，改代码时必须同步维护，
否则文档与代码、图与系统会悄悄背离：

1. **契约文档的函数签名 ↔ 实际代码**
   改了任何函数签名（参数、返回类型），回 DATA-INTERFACE-CONTRACT.md 第二部分同步。
   v1.0 交付时已逐字核对一致。

2. **拓扑图数据 ↔ 真实系统**
   SYSTEM-TOPOLOGY.html 的 `NODES`/`EDGES`/`BOUNDARIES` 三个数组是手写的，
   锁定在 2026-06-16。系统演进（v2 落地、本子系统接通真实环境、状态 pending→ok）后，
   回去改这三个数组 + 节点 status 字段。

3. **本 INDEX 的文件清单 ↔ 实际文件**
   增删文件时回来改第一节的目录树。

这三点正是本子系统将来能自动兜住的事——一旦它接通真实环境，
设计文档自身就能纳入溯源，"某个接口签名从哪版改到哪版、谁改的"可被查询。
**这套文档，是文档溯源系统自己第一个该溯源的对象。**
