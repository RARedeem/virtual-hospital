## 一、模块全景与数据流

系统由五个 Python 模块 + 一个 SQL schema 构成，单向数据流，无循环依赖：

```
  *.md 文件
     │
     ▼  splitter.split_markdown()       切分：md → 单元列表
  list[Unit]
     │
     ▼  embedder.embed_text()           向量化：每单元 → 768维（code_block 除外）
  list[NewUnit]（带 embedding）
     │
     ▼  lineage.resolve_lineage()       血缘判定：新旧比对 → 修改/新增/删除
  list[LineageDecision] + deleted_ids
     │
     ▼  provenance_ingest.ingest()      落库：写 units 表，维护血缘链
  doc_provenance.units（DB）
     │
     ▼  query.{history,search}()        读出：血缘历史 / 语义检索
  溯源结果
```

依赖方向：`ingest` 依赖 `splitter`+`embedder`+`lineage`；`query` 依赖 `embedder`。
`splitter`/`embedder`/`lineage` 三者互不依赖，可独立测试（已验证：lineage/splitter 纯逻辑单测通过，无需 DB/Ollama）。

## 二、模块详述

### 2.1 splitter（切分器）

**职责**：把 Markdown 文本切成段落级单元。对应架构文档第四节决策"段落级，不切句级"。

**为什么段落级而非句级**：中文句子边界识别不稳定（标点混用、英文术语夹杂），切到句级会让血缘链频繁断裂。段落是溯源精度与可维护性的平衡点。

**内部运转**：单遍扫描，行级状态机。维护一个 `para_buf` 累积普通段落行，遇到边界（空行/标题/列表/代码围栏/分隔线）时 `flush_para` 收口。四类单元的识别优先级：

1. 代码围栏（```/~~~）→ 整块吞入，保留围栏内原文，type=code_block
2. 空行 → 段落边界，收口当前段落
3. 分隔线（`---`/`***`/`___`）→ 跳过不入库（用户裁定）
4. 标题（`#`）→ 单独成 heading 单元
5. 列表项（`- `/`* `/`N.`）→ 单独成 list_item，续行并入同项
6. 其余 → 累积进 para_buf

**关键决策点**：
- **分隔线检测正则 `^\s*([-*_])\s*(\1\s*){2,}$`**：要求整行只有重复符号，故 `- 文本`（列表项）不会被误判为分隔线。检测顺序上分隔线在列表之前，因二者前缀符号重叠（`-`），必须靠"有无后续文本"区分。
- **代码块整块成单元、不向量化**：代码无语义检索价值（架构第四节），且代码内的 `#` 不是标题（已回归验证：`# 注释` 在代码块内不被误切）。

**已验证边界**：连续列表项、代码块内含 `#`、分隔线跳过——三组回归全通过。

### 2.2 embedder（向量化封装）

**职责**：调 Ollama 的 nomic-embed-text:v1.5，把文本转 768 维向量。

**为什么独立成模块**：向量化是 ingest（血缘判定要算相似度）和 query（语义检索要算查询向量）的共同依赖。抽出来避免两处各写一份，也便于将来换 embedding 模型时只改一处。

**内部运转**：单条 POST 到 `/api/embeddings`，带重试。

**关键决策点**：
- **NO_PROXY 强制绕代理**：本机有 VPN/代理（TODO-NEXT B2 记录），本地 Ollama 调用必须绕开，否则请求被代理劫持。模块导入时即 `setdefault` 写入环境变量，且 httpx Client 用 `trust_env=False` 双保险。这是对齐旧管道 ~/ebm-ai-pipeline 的实践。
- **冷启动重试**：nomic 模型冷启动可能返回空响应（已知现象，HANDOFF/TODO 多次记录模型冷启动返回空）。重试 3 次，退避 `2.0 * attempt` 秒，给模型加载留时间。
- **维度硬校验**：响应向量必须恰好 768 维，否则视为失败重试。防止模型版本不符或响应损坏静默入库。

**对齐风险（落地核对）**：`OLLAMA_HOST` 默认 `127.0.0.1:11434`；容器内运行须改指向 vh-ollama 的 Docker 内网地址。

### 2.3 lineage（血缘判定）

**职责**：架构文档第五节的核心——判定新版每个单元是"修改了某旧单元"还是"全新增加"，并找出被删除的旧单元。这是"向量化的核心用武之地"。

**为什么向量判定而非文本 diff**：段落改写后字面可能大变（换词、调序），但语义相近。文本 diff 会把改写误判为"删旧+增新"，丢失血缘连续性。向量相似度捕捉语义延续，才能正确串起一句话的演化链。

**内部运转**：三步——
1. 算所有 (新单元, 旧单元) 候选对的余弦相似度。
2. 按相似度降序贪心匹配：每对中若新旧都未被占用且相似度 > 阈值，锁定为"修改"。
3. 未匹配上的新单元 → 新增；未被匹配的旧单元 → 删除。

**关键决策点**：
- **0.85 阈值**（用户裁定沿用为起点）：> 0.85 判修改，≤ 0.85 判新增。架构第五节明确"实测后校准"。
- **一对一贪心匹配**（文档未覆盖，本实现补充并已报备）：防止两个新单元都匹配同一旧单元导致血缘分叉。贪心保证每个旧单元至多被一个新单元继承，符合"血缘链=线性演化"语义。这是本模块唯一偏离文档字面的设计，若预期一句可分裂成多句需回来改判。
- **code_block 直接判新增**：无 embedding，不参与向量匹配，一律 version=1 新增。
- **position 位置加权**：架构第五节列为"已知风险缓解"（段落大改相似度跌破阈值时，用位置邻近性补偿）。默认 `POSITION_WEIGHT=0`（关闭），留参数待实测真实修改模式后调，避免未验证就引入偏差。

**已验证行为**：modify / add / code_block-add / delete 四类断言全通过。

### 2.4 provenance_ingest（写入主流程）

**职责**：编排 splitter→embedder→lineage→DB，把一次文档摄取落库。对应架构第六节写入流程。

**为什么方案B手动触发**：架构第六节给了两个触发方案——A（git hook 自动）、B（手动脚本）。用户裁定方案B起步，理由是简单可控，稳定后再考虑自动化。命令形如 `python provenance_ingest.py <file> --editor=claude`。

**内部运转**：单事务包裹全流程——
1. 文档登记（按 filename 幂等，不存在则建）。
2. 取该文档现存 current 单元作血缘判定基准。
3. 调 lineage 得决策。
4. 按决策写库：修改→旧版 is_current=false + 插新版链 parent；新增→新 lineage_id；删除→旧版 is_current=false。

**关键决策点**：
- **全程单事务**：血缘链的"旧版下线+新版上线"必须原子,中途失败回滚,避免血缘链断裂或出现两个 current。
- **lineage_id 用独立 gen_random_uuid()**：与 unit_id 解耦,语义更清晰（血缘链 id ≠ 某个具体版本 id）。
- **editor 字段调用方传入**：架构要求区分 user/claude/claude-code 修编人,由 CLI `--editor` 显式指定,代码层 CHECK 约束三选一。
- **每请求新建连接**：与 orchestrator 现有风格一致。连接池是 TODO-NEXT B3 待优化项,不在本子系统范围,不提前引入。

### 2.5 query（溯源查询）

**职责**：读出溯源结果。两种查询对应用户初心的两个用例。

**两个查询的设计意图**：
- **history（血缘历史）**：用户初心"法律声明条款从原始到法律文书版,经历了哪些修改、谁改的、何时改的"。支持两种入口：按 lineage_id 直查,或按内容关键词先定位 current 单元再回溯整链。
- **search（语义检索）**：架构第五节用例"约束B在哪些地方被提到过"。用 pgvector 余弦距离检索 current 单元。

**关键决策点**：
- **history 按内容定位**：先用 ILIKE 找含关键词的 current 单元,取其 lineage_id,再拉整条链按 version 排序。让用户不必先知道 lineage_id 就能查历史。
- **search 只检索 current**：语义检索面向"现在文档里在哪提到",历史版本不参与,避免旧版污染检索结果。code_block 无 embedding 自然排除。
- **隔离保证**：所有查询 schema 限定 doc_provenance,JOIN 不触及 knowledge_base。这是架构第二节"与医疗库物理隔离"在查询层的落实——医疗检索绝不能检索到文档修改史。

## 三、跨模块的三条贯穿性设计

### 3.1 隔离（架构第二节,约束级）
全链路 schema 锁定 doc_provenance:schema 写在 SQL DDL、写在 ingest 的 INSERT、写在 query 的 FROM。三层都不出现 knowledge_base,向量空间不交叉。理由与约束 B 同源——不同性质的知识分库,互不串扰。

### 3.2 旧版永不删除
"删除"语义是 is_current=false,记录物理保留。血缘链完整可溯,这是溯源系统的根本承诺。任何模块不得物理 DELETE units 记录。

### 3.3 决策可追溯
每个偏离架构文档字面的实现决策（分隔线跳过、表格整块、一对一匹配、位置加权关闭）都在代码注释 + README + 本文档三处留痕,标注"用户裁定"或"文档未覆盖、已报备"。符合架构第十二节"文件化:关键决策写进文档,不靠记忆"。
