# 问诊实验台 interview-lab

与生产系统**完全隔离**的沙盒，专门实验"**小模型 + 临床框架 + 结构化采集**"，
不碰 orchestrator/postgres/authentik，纯标准库，独立端口 8010，仅共享 vh-ollama 推理。

## 交互形态：会员注册式表单（2026-06-25 改版）
不再是 chat 一问一答，而是把整张病史表**一次性铺开**（像网站的会员注册页），
患者**自由选序填写**；多数字段给可点选项（chips），也可「＋其他」自由补充。
- **症状包天生 JSON**：右栏实时把表单状态渲染成 `症状包(JSON)`，可复制/下载。
- **AI 只做动态追问**：填好主诉后点「🩺 医生追问」，模型看一眼已填内容，
  给相关字段追加候选、新增追问字段、并给提醒（POST `/augment`，强制 JSON）。
- **生成症状包**：点「生成症状包 ▸」让模型把已填内容规整成最终 JSON
  （含 `阳性发现` / `建议补充`，POST `/package`，强制 JSON）。
- **佐证材料上传 + OCR**：表单底部可上传检查报告/化验单/影像（点击或拖拽，可多选，单份≤25MB）。
  文件 base64 走 POST `/upload` 落盘到 `uploads/<uuid><ext>`，GET `/uploads/<id>` 可点开预览。
  - **图片**自动过 `glm-ocr`（复用 vh-ollama，非评估用途，不破坏隔离）抽文字，~0.3–1s；
    识别文本回填到可编辑框（可手动校正/补录），并进症状包 `佐证材料[].识别文本`。
    glm-ocr 小模型会循环复读 → 后端用 `stop:["```"]`+去重截断根治（见 `clean_ocr`）。
  - **PDF**：先取文本层(`pdftotext`，数字版报告秒出无损)；扫描件则栅格化(`pdftoppm`)逐页过
    glm-ocr。走 poppler CLI（subprocess，不引第三方 Python 依赖），最多 8 页，单份文本截断 8000 字。
  - **纯文本**直接解码入库；其它格式标「未支持·未OCR」，关键值填备注。
  - 症状包 JSON 自动带 `佐证材料`(文件名/大小/链接/备注/识别文本)，`/package` 生成时即读到报告内容。
  ⚠ 上传件可能含 PHI：`uploads/` 已 gitignore。

## 症状结构 = 外挂 JSON 数据集（**禁止硬编码每科字段**）
临床字段一律放在 `datasets/*.json`，`lab.py` 只做加载+组装，**不含任何科室字段表**。
- `datasets/_base.json` —— 各科共享骨架：主诉、现病史通用要素、既往史、用药、家族史。
- `datasets/<code>.json` —— 单科：`code` / `name` / `order` / `assoc`(伴随表现候选) /
  `sections`(按器官系统·专科领域铺开的分节字段)。文件名即科室 code。
- **新增/修改一个专科 = 改 JSON，零改代码**；科室下拉由 `/depts` 扫目录自动生成。
- 每次请求**实时读盘**（配合 `Cache-Control: no-store`）→ 改完刷新即生效，无需重启。

字段对象：`{key, label, type, options?, placeholder?}`，`type ∈ text|single|multi`。
组装流程在代码里（主诉→现病史→既往/用药/家族→各专科领域），**字段内容全来自数据集**。

## 启动（在你自己的终端）
```bash
python3 interview-lab/lab.py      # 监听 127.0.0.1:8010
```
浏览器开 **http://localhost:8010**。换问诊点「＋ 新问诊」。
⚠ 别在生产 /assess 跑评估时同时用（抢 GPU）。
依赖：Python 纯标准库；**PDF 佐证件需系统装 poppler-utils**(`pdftotext`/`pdftoppm`，本机已装)；
图片/扫描件 OCR 用 vh-ollama 的 `glm-ocr`。

## 实验的三个问题（对应讨论）
1. **小模型够不够**：下拉随时切模型，看追问/规整质量。
   约束A 合规小候选：`gemma4:e4b`(谷歌9.6G,默认) / `charlestang06/openbiollm`(医学5.7G) / `meditron:7b`。
   ⚠ Qwen/HuatuoGPT 等大陆机构模型仅供对照、**不可上线**（UI 已标「⚠大陆」）。
2. **结构前移**：框架直接长在表单字段上 → 结构化在采集阶段就完成，模型只补缺、做规整。
3. **临床框架在数据集里迭代**：改 `datasets/*.json` 即改框架（按器官系统/专科领域组织，
   勿"以偏概全"——非典型/无症状/偶然发现也要有入口）。代码侧 `schema_for()` 只组装，不写字段。
   验证"框架扛结构 → 小模型够用 + 症状包天生结构化"。

## 已验证（2026-06-25）
`gemma4:e4b` 实测：`/augment` 15s 出有效 JSON（按尿频主诉追加血尿/疼痛部位）、
`/package` 8s 把已填内容规整成含 `阳性发现`/`建议补充` 的症状包 JSON。
→ 印证：结构前移进表单后，小模型只需补缺与规整，又快又稳。
