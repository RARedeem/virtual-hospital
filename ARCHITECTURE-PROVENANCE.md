> 版本：v1.0（设计定稿，实现入队，主线优先）
> 日期：2026-06-16
> 状态：架构已定，待实现（不阻塞虚拟医院 v1/v2 主线）
> 定位：虚拟医院项目的**支撑子系统**，非主线功能

---

## 一、需求来源与本质

**用户初心**：对项目设计文档（ARCHITECTURE.md、TODO-NEXT.md、声明条款等）实现细粒度的变更溯源——每个文本单元都能追溯其修改历史、修编人、版本号。典型用例："法律声明条款从原始语料到法律文书版，经历了哪些修改、谁改的、何时改的。"

**本质澄清**：用户最初称之为"向量化"，但严格说——
- 溯源能力 ≠ 向量化。向量化（embedding）本身不记录历史。
- 真正的实现是**句子/段落级版本化数据库**（记录 who/when/what）。
- **向量化在本系统中承担两个角色**：
  1. 检索层——语义搜索文本单元（"约束B在哪些地方被提到过"）
  2. 血缘识别层——判定新旧版本间"修改 vs 新增"（见第五节）

用户的向量化直觉是对的，只是它的正确落点是"检索 + 血缘识别"，而非"溯源本身"。

---

## 二、与医疗知识库的隔离（不可违反）

**决策（架构师定，约束级）**：文档溯源系统与医疗 RAG 物理隔离。

- 医疗知识库：schema `knowledge_base`（guideline_chunks 等）
- 文档溯源库：schema `doc_provenance`（独立，新建）

**理由**：医疗检索（如前列腺指南）绝不能检索到"ARCHITECTURE.md 第3节的修改史"。混用会污染医疗评估。此隔离与约束 B（指南来源隔离）同源——不同性质的知识必须分库，互不串扰。

两者共用同一套基础设施（pgvector + postgres + nomic-embed），但 schema 隔离、检索查询隔离、向量空间隔离。

## 三、数据模型

### 3.1 schema: doc_provenance

**表一：documents（文档登记）**

| 字段 | 类型 | 说明 |
|------|------|------|
| doc_id | uuid PK | 文档唯一标识 |
| filename | text | 如 ARCHITECTURE.md |
| created_at | timestamptz | 首次纳入溯源的时间 |

**表二：units（文本单元 + 版本）**

| 字段 | 类型 | 说明 |
|------|------|------|
| unit_id | uuid PK | 单元版本唯一标识（每个版本一条记录） |
| doc_id | uuid FK | 所属文档 |
| lineage_id | uuid | 血缘链标识（同一句的所有版本共享此 id） |
| version | int | 该单元的版本号（从 1 递增） |
| content | text | 文本内容（段落/列表项/标题） |
| unit_type | text | paragraph / list_item / heading / code_block |
| position | int | 在文档中的顺序位置 |
| editor | text | user / claude / claude-code |
| parent_unit_id | uuid FK nullable | 上一版本的 unit_id（血缘指针） |
| created_at | timestamptz | 此版本产生时间 |
| is_current | bool | 是否为该血缘链的当前版本 |
| embedding | vector(768) | nomic-embed 向量（检索 + 血缘判定用） |

**血缘机制**：
- 同一个 lineage_id 串起一句话的所有历史版本。
- 改一句 → 新增一条 units 记录，version+1，parent_unit_id 指向旧版，旧版 is_current=false，新版 is_current=true。
- 旧版**永不删除**，溯源链完整保留。

### 3.2 溯源查询示例（用户初心的落地）

```sql
-- 查"法律声明"这句的完整进化史
SELECT version, editor, created_at, content
FROM doc_provenance.units
WHERE lineage_id = (
  SELECT lineage_id FROM doc_provenance.units
  WHERE content ILIKE '%不构成医学诊断%' AND is_current = true
)
ORDER BY version;
```

## 四、文本单元切分（决策：段落级）

**决策（架构师定）**：按段落 / 列表项 / 标题切分，**不切到标点/句子级**。

**理由**：
- 中文句子边界识别不稳定（标点混用、英文术语夹杂），切到句级血缘链易断。
- 段落级是"溯源精度"与"可维护性"的最佳平衡点。
- 一个 Markdown 段落、一个列表项、一个标题各为一个单元。

**切分规则**：
- 空行分隔的文本块 = 一个 paragraph 单元
- `- ` / `* ` / `N.` 开头的行 = 一个 list_item 单元
- `#` 开头的行 = 一个 heading 单元
- 代码块（三反引号包裹）整块作为一个 unit，type=code_block，不向量化（无语义检索价值）

---

## 五、血缘判定（修改 vs 新增）——向量化的核心用武之地

**问题**：文档更新后，如何判定"这个新单元是某旧单元的修改"还是"全新增加"？这决定了血缘链能否正确延续。

**决策（架构师定）**：用向量相似度判定。
- 对新版每个单元，在旧版同文档单元中找向量相似度最高者。
- 相似度 > 0.85 → 判为"修改"：新单元继承该旧单元的 lineage_id，version+1，parent_unit_id 指向它。
- 相似度 ≤ 0.85 → 判为"新增"：新建 lineage_id，version=1。
- 旧版中未被任何新单元匹配上的 → 判为"删除"：旧单元 is_current=false，不再有后继（血缘链终止，但历史保留）。

**阈值 0.85 为起点，实测后校准**。这是用户"向量化"直觉的正解落点——向量在此做血缘识别，而非医疗检索。

**已知风险（待实现时验证）**：
- 段落大改时相似度可能跌破 0.85，被误判为"新增+删除"而非"修改"。缓解：可辅以位置（position）邻近性加权。实现时观察真实文档的修改模式再调。

## 六、写入流程

```
文档更新（git commit 后触发，或手动调用）
  → 读取新版 md
  → 按第四节规则切分成单元
  → 每个单元 nomic-embed 向量化
  → 与旧版单元做血缘判定（第五节）
  → 修改：version+1，链接 parent；新增：新 lineage；删除：标记旧单元
  → 写入 doc_provenance.units
  → editor 字段由调用方传入（user/claude/claude-code）
```

**触发方式（待定，实现时选）**：
- 方案A：git post-commit hook 自动触发（改动即溯源）
- 方案B：手动跑脚本 `python provenance_ingest.py <file> --editor=claude`
- 倾向方案B起步（简单、可控），稳定后再考虑A。

## 七、与 git 的关系（不是替代，是互补）

| 能力 | git | 本溯源系统 |
|------|-----|-----------|
| 文件级/行级 diff | 强 | — |
| 谁、何时、改了哪几行 | ✓ | ✓ |
| 段落级血缘链（一句话跨多次修改的完整演化） | 难 | ✓ |
| 语义检索（"约束B在哪被改过"） | ✗ | ✓ |
| 修编人区分 user/claude/claude-code | 靠 commit author | 字段级 ✓ |

**定位**：git 管文件版本快照（已在用，继续用）；本系统管段落级血缘 + 语义溯源检索。二者互补，非替代。git 是基线，本系统是精细溯源层。

## 八、实现优先级（用户决策：入队，主线优先）

**用户拍板**：先写架构文档钉死设计（本文档），实现排进队列，虚拟医院主线（v1 收尾 / v2 双盲）优先。

**实现条件**：当主线推进到一个稳定节点、或文档溯源需求变得迫切时，再由 Claude Code 照本文档实现。

**实现拆解（待启动时按序）**：
1. 建 doc_provenance schema + 两张表
2. 写切分器（md → units）
3. 写向量化 + 血缘判定逻辑
4. 写入流程脚本（方案B手动起步）
5. 写溯源查询接口（按 lineage 查历史）
6. 用真实文档（ARCHITECTURE.md 的 git 历史）回测：能否正确还原"声明条款"的进化史

**验证标准**：能查出任意一个段落的完整修改史（版本号、修编人、时间、每版内容），且医疗检索不受任何污染。

## 十、一句话总结

用户的"向量化溯源"初心 = 句子级版本化数据库（溯源）+ 向量检索（检索与血缘识别）。向量化不独自做溯源，但它是溯源系统的检索入口和血缘识别器。本系统与医疗库严格隔离，与 git 互补，段落级粒度，设计已定，实现入队。
